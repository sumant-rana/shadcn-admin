# Release Highlights v2.3.0 — `release-highlights` review experience

This release ships a polished, customer-facing **Release Highlights** review page at `/release-highlights` in the shadcn-admin demo. It is purpose-built so internal reviewers can answer four questions in under a minute: **what changed, who it matters to, what the risks are, and which supporting assets are ready**.

## Summary

A new route is added to the authenticated shell of shadcn-admin that consolidates per-release review context into one screen — hero, filters, highlight cards, a walkthrough tile, a readiness panel, and a post-launch pulse panel. It renders entirely from local fixture data, so it is safe to demo, screenshot, and record without any backend dependency.

## Goal

Give internal reviewers (product, docs, field, security, enablement) a single place to triage a release so they can decide what to approve, edit, or escalate, and so downstream automation (docs, blog, video, social) has a single grounded source of truth to cite.

## Audience

| Audience | Why they care |
|---|---|
| **Admin** | Launch readiness, rollout gating, who owns what |
| **Security** | Risk surface, blocked claims, mitigations |
| **Reporting** | Adoption, usage, errors, support signal in the first 7 days |
| **UI** | Visual and IA changes, copy review, accessibility |

## What ships

### `/release-highlights` route

A new authenticated route registered through the existing TanStack route tree. It is reachable from the sidebar and renders without any new backend dependencies.

### Visual system

The page reuses the established shadcn-admin design system: card, badge, button, tabs, separator, skeleton, scroll-area primitives; the dark theme; the standard page header and spacing scale. No new design tokens are introduced.

### Hero section

A hero block at the top of the page shows:

- Release title and one-line summary
- Version (`v2.3.0-release-highlights`) and published date
- Readiness status pill (`Go`, `Hold`, or `No-go`)
- Owner and review-cycle metadata

### Audience and risk filters

Two filter rows sit directly under the hero:

- **Audience:** `Admin`, `Security`, `Reporting`, `UI` (multi-select chips)
- **Risk:** `Low`, `Medium`, `High` (multi-select chips)

Filters update the visible highlight cards client-side. Filter state is reflected in the URL so reviewers can share a focused slice.

### Highlight cards

Each card shows the same six fields so review is consistent:

- Title
- Short customer-facing description (one to three sentences, plain language)
- Audience badge (matches the filter chip)
- Risk badge (`Low` / `Medium` / `High`, color-coded)
- Evidence/source indicator (PR link, doc link, or test artifact)
- Recommended reviewer action (`Approve`, `Request edits`, `Escalate`)

Seeded demo data covers all four audiences and all three risk levels.

### Watch walkthrough tile

A prominent `Watch walkthrough` tile in the right column has three explicit visual states:

- **Empty** — "No walkthrough generated yet" placeholder
- **Generating** — animated skeleton + "Walkthrough rendering…" copy
- **Ready** — embedded player + permalink + caption

The tile reads `window.latestWalkthroughVideoUrl` when present so the video agent can populate it post-render. The empty state is the default during initial replay.

### Readiness panel

A panel surfaces the go/no-go signal with four sub-sections:

- Overall go/no-go status with rationale
- Blocked claims (citation guard hits, missing evidence)
- Test evidence (Playwright runs, unit coverage delta)
- Launch risks (named risks with owners and mitigation links)

### Post-launch pulse panel

A panel summarizes early signal for the release:

- 24h and 7d active-user counts vs. prior release
- Adoption funnel (visited → engaged → completed)
- Error rate (5xx, client-side exceptions) with delta
- Support signals (open tickets tagged to the release)

For the demo, this panel is seeded with realistic synthetic numbers.

## Risk and readiness metadata

- **Overall risk:** Medium
- **Primary audience:** Admin reviewers preparing the launch
- **Secondary audiences:** Reporting, Security, UI
- **Human review required:** Yes (combines new navigation, generated copy, and a video walkthrough)
- **Known demo risk:** The walkthrough tile renders the empty state on first replay until the video agent finishes
- **Blocked claims:** None at publish time
- **Rollback plan:** Hide the sidebar entry behind a feature flag; the route remains routable but unlinked

## Acceptance criteria

- `/release-highlights` renders without backend dependencies using local fixture data.
- Audience and risk filters update the visible cards immediately.
- The walkthrough tile clearly shows empty, generating, and ready states.
- The page is suitable for screenshot and video capture in an automated release demo.
- The page is responsive and visually polished at desktop and tablet widths.
- All interactive controls have accessible labels and semantic headings.

## Demo and video target

For automated video capture, point the video agent at:

```
AUTOPILOT_VIDEO_TARGET_URL=http://host.docker.internal:5173/release-highlights
```

The walkthrough script should narrate, in order:

1. Hero + readiness status
2. Audience and risk filters (toggle one of each)
3. A representative highlight card per audience
4. The walkthrough tile (empty state is expected)
5. Readiness panel highlights
6. Post-launch pulse panel summary

## Validation notes

- Verified fixture data covers all four audiences and all three risk levels.
